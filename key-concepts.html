Domain 1: Cloud Concepts

Task Statement 1.1: Define the benefits of the AWS Cloud.

What this means: You must connect business goals to cloud outcomes: faster delivery, global scale, resilience, and lower operational overhead.

What you need to know: The six key benefits of the AWS Cloud are:

  • Trade fixed expense for variable expense: Replace large upfront infrastructure investment with pay-as-you-go usage.
  • Benefit from massive economies of scale: AWS global scale lowers cost per unit and makes advanced capabilities accessible to any company size.
  • Stop guessing capacity: Scale resources up or down based on real demand instead of overprovisioning or underprovisioning.
  • Increase speed and agility: Deploy faster, shorten time to market, and respond quickly to business changes.
  • Stop spending money to run and maintain data centers: AWS manages physical infrastructure so teams can focus on business value.
  • Go global in minutes: Use AWS Regions and global infrastructure to launch workloads internationally without building datacenters.

Exam clue: If the question emphasizes speed, flexibility, or uncertain demand, cloud elasticity and managed services are usually key.

Task Statement 1.2: Identify design principles of the AWS Cloud.

What this means: Understand AWS Well-Architected pillars and how they influence technical decisions.

What you need to know:

  • Operational Excellence: run and improve systems continuously.
  • Security: protect data, systems, and identities at every layer.
  • Reliability: recover quickly and handle disruptions.
  • Performance Efficiency: select right resources and evolve with demand.
  • Cost Optimization: avoid waste and align spend with business value.
  • Sustainability: reduce environmental impact through efficient workloads.

Exam clue: Questions often test pillar differentiation, for example reliability vs performance efficiency or security vs operational excellence.

Task Statement 1.3: Understand the benefits of and strategies for migration to the AWS Cloud.

What this means: You should identify why organizations migrate and which migration pattern fits each scenario.

What you need to know:

  • Migration drivers: cost reduction, agility, modernization, and risk reduction.
  • AWS Cloud Adoption Framework (CAF) links business, people, governance, platform, security, and operations to migration outcomes.
  • Migration patterns include rehost, replatform, and refactor; CLF-C02 tests recognition, not deep engineering steps.
  • AWS Snowball and AWS DMS appear in scenarios with data movement constraints.

Exam clue: Large data transfer with limited bandwidth usually points to Snowball; ongoing DB migration with low downtime often points to DMS.

Task Statement 1.4: Understand concepts of cloud economics.

What this means: Compare financial models and identify how cloud changes cost structure and optimization options.

What you need to know:

  • Fixed costs (datacenter hardware) vs variable costs (pay-as-you-go resources).
  • Rightsizing reduces waste by aligning resource size with actual demand.
  • Economies of scale in cloud can reduce unit cost versus on-premises operations.
  • Automation reduces operational overhead and human error.
  • Licensing models: BYOL versus license included.

Exam clue: If a question asks for lowering idle spend, look for rightsizing, Auto Scaling, or pricing model adjustments.

Domain 2: Security and Compliance

Task Statement 2.1: Understand the AWS shared responsibility model.

What this means: Distinguish security responsibilities of AWS versus customer, depending on service type.

What you need to know:

  • AWS secures the cloud infrastructure (physical facilities, hardware, managed service platform).
  • Customer secures what runs in the cloud (data, identities, configurations, access control).
  • Responsibility shifts by service model: EC2 gives customer more control and more responsibility than Lambda.
  • Managed databases like RDS reduce OS patching burden compared to self-managed EC2 database hosts.

Exam clue: If the question asks who patches the guest OS in EC2, that is customer responsibility.

Task Statement 2.2: Understand AWS Cloud security, governance, and compliance concepts.

What this means: Map compliance and security requirements to the right AWS capabilities and evidence sources.

What you need to know:

  • AWS Artifact provides compliance reports and agreements.
  • Encryption in transit protects data over networks; encryption at rest protects stored data.
  • CloudTrail captures API activity for auditing; CloudWatch monitors metrics and logs.
  • AWS Config tracks resource configuration changes and compliance state over time.
  • Audit Manager helps map controls to evidence for audits.

Exam clue: Questions about proving compliance or retrieving audit evidence often lead to Artifact, CloudTrail, Config, or Audit Manager.

Task Statement 2.3: Identify AWS access management capabilities.

What this means: You must design least-privilege access using IAM identities, policies, federation, and MFA.

What you need to know:

  • IAM users, groups, roles, and policies are core identity building blocks.
  • Least privilege means granting only permissions required for the task.
  • MFA is critical for privileged access, especially root user and admin paths.
  • IAM Identity Center enables centralized workforce access across AWS accounts and apps.
  • Secrets Manager and Systems Manager Parameter Store are preferred over hardcoded credentials.

Exam clue: If access must be temporary or cross-account, IAM roles are usually preferred over long-lived access keys.

Task Statement 2.4: Identify components and resources for security.

What this means: Recognize AWS-native services that detect, protect, and respond to threats.

What you need to know:

  • GuardDuty for threat detection and suspicious behavior findings.
  • AWS WAF for Layer 7 web filtering, AWS Shield for DDoS protection.
  • Security Hub centralizes security posture and findings across services.
  • Firewall Manager provides centralized policy management for multi-account security controls.
  • Trusted Advisor includes security and cost recommendations.

Exam clue: Questions about centralized visibility and aggregation of security findings often point to Security Hub.

Domain 3: Cloud Technology and Services

Task Statement 3.1: Define methods of deploying and operating in the AWS Cloud.

What this means: Select the right interface and operating model for the workload and team maturity.

What you need to know:

  • AWS Management Console is best for interactive/manual tasks.
  • CLI, APIs, and SDKs are best for automation and repeatability.
  • Infrastructure as Code (for example CloudFormation) enables versioned, consistent deployments.
  • Cloud, hybrid, and on-premises models differ by where workloads run and who manages infrastructure.

Exam clue: If the requirement is consistency and repeatability across environments, prefer IaC.

Task Statement 3.2: Define the AWS global infrastructure.

What this means: Understand Regions, Availability Zones, and edge locations and how they improve resilience and latency.

What you need to know:

  • Region is a geographic area; each Region has multiple Availability Zones.
  • Availability Zones are physically separate, reducing shared failure risk.
  • Use multi-AZ for high availability and fault tolerance.
  • Use multi-Region for disaster recovery, sovereignty needs, and global latency goals.
  • Edge locations improve content delivery and edge processing performance.

Exam clue: High availability usually means multiple AZs; business continuity across major outages usually means multiple Regions.

Task Statement 3.3: Identify AWS compute services.

What this means: Match workload type to compute model: virtual machines, containers, or serverless.

What you need to know:

  • Amazon EC2 for maximum control over operating system and runtime.
  • Lambda for event-driven serverless code with per-invocation billing.
  • ECS/EKS for container orchestration; Fargate for serverless containers.
  • Auto Scaling adjusts capacity dynamically; Elastic Load Balancing distributes traffic.
  • Instance family choice matters (compute-optimized, memory-optimized, storage-optimized, and so on).

Exam clue: Unpredictable traffic with minimal ops usually favors serverless + Auto Scaling + load balancing patterns.

Task Statement 3.4: Identify AWS database services.

What this means: Select database type by access pattern, consistency needs, and operational overhead tolerance.

What you need to know:

  • RDS/Aurora for relational workloads requiring SQL and structured schema.
  • DynamoDB for key-value/document NoSQL workloads and millisecond latency at scale.
  • ElastiCache for in-memory caching and ultra-low-latency reads.
  • AWS DMS and AWS SCT support migration from on-premises or other engines.
  • Managed databases reduce patching and backup overhead compared to self-managed EC2 databases.

Exam clue: If the requirement says managed relational with minimal admin, RDS or Aurora is a common answer.

Task Statement 3.5: Identify AWS network services.

What this means: Know core VPC building blocks and connectivity choices.

What you need to know:

  • VPC components include subnets, route tables, internet/NAT gateways, and security boundaries.
  • Security groups are stateful instance-level filters; network ACLs are subnet-level filters.
  • Route 53 provides DNS routing and health-based traffic policies.
  • AWS VPN uses encrypted tunnel over internet; Direct Connect provides dedicated private connectivity.

Exam clue: Requirement for consistent, private, high-throughput link to AWS usually points to Direct Connect.

Task Statement 3.6: Identify AWS storage services.

What this means: Match object, block, and file storage to workload behavior and cost/performance goals.

What you need to know:

  • Amazon S3 for durable object storage, backups, and data lakes.
  • S3 storage classes differ by access frequency, retrieval speed, and price.
  • Amazon EBS for block storage attached to EC2; instance store for ephemeral local storage.
  • Amazon EFS and Amazon FSx for shared file workloads.
  • Storage Gateway extends on-premises environments with cloud-backed storage patterns.
  • Lifecycle policies and AWS Backup support long-term cost control and protection strategy.

Exam clue: If data is accessed rarely but must be retained, S3 lifecycle with infrequent/archive class is common.

Task Statement 3.7: Identify AWS AI/ML and analytics services.

What this means: Recognize services by business outcome: predictions, NLP/chat, streaming analytics, BI reporting, and data preparation.

What you need to know:

  • SageMaker for building, training, and deploying ML models.
  • Lex for conversational chatbots and voice interfaces.
  • Kendra for intelligent enterprise search use cases.
  • Athena for SQL queries on S3 data without provisioning servers.
  • Kinesis for streaming ingestion and real-time processing patterns.
  • Glue for data integration/ETL and cataloging; QuickSight for dashboards and BI.

Exam clue: If analysts need ad-hoc SQL on S3, Athena is usually the direct answer.

Task Statement 3.8: Identify services from other in-scope AWS service categories.

What this means: CLF-C02 expects broad service awareness across integration, business applications, developer tools, end-user computing, frontend/mobile, and IoT.

What you need to know:

  • EventBridge, SNS, and SQS for eventing, pub/sub messaging, and queue decoupling.
  • Amazon SES for scalable email; Amazon Connect for contact center capabilities.
  • CodeBuild and CodePipeline for CI/CD workflows; X-Ray for tracing and troubleshooting.
  • AppStream 2.0 and WorkSpaces for end-user virtual desktop and app streaming scenarios.
  • AWS Amplify and AppSync for frontend/mobile backends and APIs.
  • AWS IoT Core for secure device connectivity and message routing.
  • AWS Support plans define response times and guidance level.

Exam clue: For asynchronous decoupling between services, SQS is usually preferred; for fan-out notifications, SNS is common.

Domain 4: Billing, Pricing, and Support

Task Statement 4.1: Compare AWS pricing models.

What this means: Choose purchasing model based on predictability, flexibility, and tolerance for interruption.

What you need to know:

  • On-Demand for short-term or unpredictable workloads.
  • Reserved Instances and Savings Plans for committed usage and lower long-term cost.
  • Spot Instances for interruptible workloads with high discount potential.
  • Dedicated Hosts/Instances for isolation, compliance, or licensing constraints.
  • Data transfer pricing differs for inbound vs outbound and inter-Region traffic.

Exam clue: Stable baseline workload generally maps to Reserved/Savings model; fault-tolerant batch jobs often map to Spot.

Task Statement 4.2: Understand resources for billing, budget, and cost management.

What this means: Track, forecast, allocate, and optimize spend using AWS financial management tools.

What you need to know:

  • AWS Budgets for threshold alerts and budget governance.
  • Cost Explorer for spend analysis, trends, and usage forecasting.
  • Pricing Calculator for pre-deployment cost estimates.
  • AWS Organizations consolidated billing for multi-account cost visibility.
  • Cost allocation tags and Cost and Usage Report (CUR) for chargeback/showback and detailed reporting.

Exam clue: Questions about forecasting and visualizing spend over time usually point to Cost Explorer.

Task Statement 4.3: Identify AWS technical resources and AWS Support options.

What this means: Know where to get official guidance, troubleshooting help, and enterprise support channels.

What you need to know:

  • AWS documentation, whitepapers, blogs, Prescriptive Guidance, and re:Post for self-service learning and troubleshooting.
  • Support plans range from basic account support to enterprise-grade technical guidance.
  • Trusted Advisor and AWS Health help identify operational, security, and cost-impacting issues.
  • AWS Partners and AWS Marketplace provide third-party solutions and implementation expertise.
  • AWS Professional Services and Solutions Architects can accelerate architecture and migration efforts.

Exam clue: If the scenario asks for fastest, business-critical technical response and architectural guidance, enterprise-level support is usually expected.
